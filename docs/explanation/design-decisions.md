# Design Decisions

## Equal-weight allocation over optimization-based allocation

Strategies and assets receive equal capital shares. No prediction is made about which strategy will outperform. This avoids overfitting to historical performance, reduces the parameter space, and makes the system robust to regime changes. The portfolio's edge comes from diversification across uncorrelated strategies rather than from optimal weighting.

## Services as separate .mq5 files over in-process threads

MetaTrader 5 does not support multi-threading within a single EA. The only way to run concurrent work is through separate MT5 service scripts. Each service (`HorizonPersistence`, `HorizonGateway`, `HorizonMonitor`) is a standalone `.mq5` file with its own `OnStart()` loop. Services communicate with the EA exclusively through the shared-memory message bus -- there are no shared variables or direct function calls between the EA and services.

## Message bus over direct HTTP calls

HTTP calls in MQL5 are synchronous and can block for seconds on timeouts or slow networks. By routing all I/O through a DLL-backed message bus with fire-and-forget publishing and asynchronous polling, the EA's `OnTimer` and `OnTradeTransaction` callbacks remain fast. The bus also provides sequence-based acknowledgment, deduplication (in the Persistence service), and queue diagnostics.

## JSON file persistence over database

MT5 has no native database access. File I/O is the only local persistence mechanism. Horizon5 serializes order state and statistics as JSON files, which are human-readable, easy to debug, and can be loaded on restart for order recovery. The Persistence service handles writes asynchronously and deduplicates rapid updates to the same file path, keeping disk I/O manageable.

## Deterministic identifiers over server-assigned IDs

Magic numbers (DJB2 hash of `symbol_assetName_strategyName`, mod 1 billion) and UUIDs (seeded hash formatted as UUID v5) are computed locally from stable inputs. This eliminates registration round-trips, survives restarts, and allows the EA and the Horizon API to independently compute matching identifiers. The EA validates magic number uniqueness on startup.

## City naming convention for strategies

Strategies are named after cities grouped by asset class: South African/Australian cities for Gold, Japanese cities for Nikkei225, American cities for S&P500. Names must be easy to pronounce and memorable. Each strategy also gets a 3-letter uppercase prefix (e.g., SYD, ADL, AUS). This convention makes strategies instantly identifiable in logs, dashboards, and conversations without needing to decode technical identifiers.

## Strategy development pipeline

The framework separates strategy discovery from strategy execution. Strategies can be developed manually, generated by automated tools, or ported from other platforms. The EA provides the infrastructure -- order management, risk sizing, persistence, monitoring -- while strategy creation follows whichever workflow the developer prefers.

## Equity-at-risk position sizing over fixed lot

Position size is calculated based on a configurable equity-at-risk percentage and the stop-loss distance, using the formula: `lots = (nav * equityAtRisk%) / (stopLossDistance * tickValue / tickSize)`. This scales position size with account equity (optionally compounded) and normalizes risk across instruments with different tick values and volatilities. A fixed-lot approach would either under-utilize capital on large accounts or over-leverage small ones.
